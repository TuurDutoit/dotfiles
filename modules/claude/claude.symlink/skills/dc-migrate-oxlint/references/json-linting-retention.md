# JSON-linting retention — scoped ESLint

Default answer when oxlint can't lint JSON but the repo has a real, existing ESLint use case. `oxfmt` formats JSON but catches only syntax; specific ESLint plugins cover gaps (`plugin:json/recommended` for duplicate keys, `eslint-plugin-i18n-json` for cross-file locale parity + ICU MessageFormat). Do **not** reimplement these — retain ESLint scoped to just that purpose.

## Detection signals

Check before finishing cleanup (step 14/16). If any apply, retention kicks in:

- **Hand-edited structural JSON where content matters** (schemas, contracts, fixtures) — retain for duplicate-key detection via `plugin:json/recommended`. Grep for JSON files outside `node_modules/`, `dist/`, `coverage/`, `package.json`, `tsconfig*.json`, `.vscode/`, autogenerated paths — if anything meaningful remains, the repo is a candidate.
- **Existing `eslint-plugin-i18n-json` + `lint-translations` script** — retain for cross-file locale parity. Applies to \~23 DC repos.
- **Other valid ESLint JSON use cases** — same retention pattern, different plugin. Apply judgement; the principle is "if the ESLint plugin covers something oxlint can't, scope it down and keep it."

If **none** apply, the cleanup in step 14 fully removes ESLint — no retention needed.

## Pattern

Identical across usages.

### ESLint version

**ESLint 9+** with flat config. The v4 releases of `eslint-plugin-json` and `eslint-plugin-i18n-json` both support ESLint 9+ via flat config. Legacy `.eslintrc.*` format is deprecated in ESLint 9 and removed in 10 — don't use it.

### One config file per repo/package

`eslint.config.json.mjs` — usage-agnostic. Multiple JSON-linting usages (schemas, translations, anything else) live as entries inside it, each keyed by its own `files` glob. Naming the file after the file-type (not the usage) keeps it stable when a repo later adds another JSON-linting need.

### Steps

1. Keep `eslint` in devDependencies (`yarn add -D eslint` grabs the current latest; lockfile pins it).

2. Keep the usage-specific plugin(s) at v4 or newer (`eslint-plugin-json@^4`, `eslint-plugin-i18n-json@^4`, etc.).

3. Create `eslint.config.json.mjs` (flat config, ESM — `.mjs` extension works regardless of the package's `"type"` field). Include one entry per JSON-linting usage this repo needs. **No** shared-config extends, **no** plugins beyond what those usages require:

   ```js
   // eslint.config.json.mjs — all JSON linting for this repo/package
   import json from 'eslint-plugin-json';
   import i18nJson from 'eslint-plugin-i18n-json';

   export default [
     // Schema dup-key detection (include if repo has hand-edited structural JSON)
     {
       files: ['src/**/*.schema.json'], // adjust glob to the repo's schema path
       ...json.configs.recommended,
     },
     // i18n translation validation (include if repo has locale JSON) — verify rule list per repo
     {
       files: ['src/languages/*.json'],
       plugins: { 'i18n-json': i18nJson },
       processor: {
         meta: { name: '.json' },
         ...i18nJson.processors['.json'],
       },
       rules: {
         ...i18nJson.configs.recommended.rules,
         // Override per repo as needed, e.g.:
         // 'i18n-json/valid-message-syntax': ['error', { syntax: 'icu' }]
       },
     },
   ];
   ```

   Include only the entries this repo needs. Add further entries for any other valid JSON-linting usage (same shape: `files` glob + rules).

4. Add dedicated yarn scripts — mirror the `lint:check` / `lint:fix` split. Pass `--config <file>` to restrict flat-config discovery to that single file:

   ```json
   "lint:check:json": "eslint --config eslint.config.json.mjs '<json-glob>'",
   "lint:fix:json":   "eslint --config eslint.config.json.mjs --fix '<json-glob>'"
   ```

   `<json-glob>` should cover every path matched by any entry in the config (e.g. `'src/**/*.json'`). With `--config`, ESLint uses only the specified file — no walk-up, no discovery.

5. **Wire into local dev** via the `check` composite (scoped script runs last so faster tools fail first):

   ```json
   "check": "yarn typecheck && yarn format:check && yarn lint:check && yarn lint:check:json"
   ```

6. **Wire into CI** — depends on the repo's CI `check` job shape:
   - **CI runs `yarn check`** (composite-style): the fold in step 5 propagates. No CI config changes.
   - **CI runs per-phase sub-scripts** (`yarn typecheck`, `yarn format:check:ci`, `yarn lint:check:ci` as separate `run:` steps — the `ci-split` pattern): add a `Lint JSON` step to the `check` job, calling `yarn lint:check:json`. For parameterised monorepo jobs (Pattern 2 / A2), gate the step on a per-workspace boolean (e.g. `has_json_lint: true/false`) and only set it on workspaces that have the script. For Lerna monorepos (Pattern 3 / A1), add a root wrapper (`"lint:check:json:ci": "lerna run lint:check:json --parallel --no-bail"`) and call that from CI.

7. **Everything else ESLint-related is still removed** per step 14 — main `eslint.config.*` / `.eslintrc.*` for JS/TS, `@datacamp/eslint-config`, every other `eslint-plugin-*` (beyond the scope plugin(s)), `eslint-config-prettier`, `eslint-plugin-prettier`.

## Net result

Main `yarn lint` is oxlint only. `yarn lint:check:json` is a minimal ESLint call covering every JSON-linting usage in the repo via a single config file. No double-install of ESLint, no shared-config pollution, no CLI drift.

## VS Code

`"eslint.enable": false` in `.vscode/settings.json` (per step 16) remains correct — ESLint runs only via the CLI script, not the VS Code extension. If a contributor wants live schema feedback locally, they can toggle the extension on in their user settings.
